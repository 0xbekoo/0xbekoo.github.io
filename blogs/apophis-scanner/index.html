Apophis Scanner: GPU and CPU Based Regex Scanner - Thu, Jan 1, 2026

Welcome to my another project! Today in this blog i will introduce my project: GPU and CPU Based Regex Scanner: Apophis Scanner.

A few weeks ago, I came across an article about performing REGEX scans using a GPU, and it really caught my attention. As I began to delve into the details, I decided to take a break from the CPU for a while and focus on this, and ultimately I developed a GPU-based scanner. In this blog, we will explore the details of my project.

Why “Apophis”? Well, In ancient mythology, Apophis was the giant serpent of chaos, known as the “Eater of the Sun.” My project does something similar—it doesn’t eat suns, but it devours bytes. Terabytes of them. At the beginning of the project, i called this project ‘Byte Eater’.

What is the Project?

This project is similar to hyperscan but the ideas are different. Essentially this project scans Regex values with the resources of GPU and CPU.

Today, this REGEX scan is performed using the CPU, and we are working to optimize the CPU for efficient operation. What if we split this process in half and try to perform it using both the CPU and GPU? Well, Apophis Scanner does exactly that.

Running a regex scan for the GPU may seem strange, but there is a lot of research on this topic, and the main subject is ‘Multi-Pattering’. Now let’s take a closer look at this:

Scanning Regex With Multi-Pattering

In a traditional scenario, if you want to scan a file for 50 different malware signatures, a standard regex engine typically iterates through the signatures. It looks for the first one, then the second, and so on. Even if it optimizes this into a tree structure (like Aho-Corasick), it involves complex memory jumps and pointer chasing. But Apophis takes a different approach. It cheats physics by using Bit-Parallelism.

According to the paper ‘Interleaved Bitstream Execution for Multi-Pattern Regex Matching on GPUs’, trying to run standard Regex algorithms on a GPU introduces two major bottlenecks that usually kill performance. These are:

1 - Control Flow Divergence (The “If-Else” Nightmare)

Standard Regex engines (based on DFAs or NFAs) are decision-heavy. They operate on logic like: “If character is ‘A’, go to State 2; otherwise, go to State 3.”

On a CPU, this is fine. But a GPU is a SIMT (Single Instruction, Multiple Threads) device. It wants thousands of threads to execute the exact same instruction at the same time. When you introduce conditional branching (if-else), some threads want to take the “True” path, while others want to take the “False” path.

The GPU cannot do both simultaneously. It has to serialize them: it pauses the “False” group, runs the “True” group, and then switches. This is called Control Flow Divergence, and it leaves most of your compute power sitting idle.

2 - Irregular Memory Access (The Bandwidth Killer)

Regex matching usually involves a “State Transition Table.” As the engine reads characters, it jumps around this table randomly to find the next state.

  • Thread 1 might need data from address 0x100.
  • Thread 2 might need data from address 0x900.

GPUs hate random access. Because they are built for Coalesced Memory Access—where threads read sequential blocks of memory (like a firehose). Random jumps cause the memory controller to stall, wasting the GPU’s massive bandwidth potential.

To solve these problems, the paper proposes a novel architecture known as BitGen. It completely abandons the “State Machine” approach in favor of a Bitstream approach Instead of navigating a complex graph of states, BitGen converts the regex logic into a stream of bits. It uses two key strategies to unlock the GPU’s full potential:

  • Bit-Parallel NFA: It replaces conditional branching (if) with bitwise operations (AND, OR, SHIFT). Since math is deterministic, every thread executes the same instructions regardless of the input data. Divergence is eliminated.

  • Interleaved Data Layout: Instead of each thread reading a separate chunk of the file, threads read data in an interleaved pattern (Thread 0 reads byte 0, Thread 1 reads byte 1…). This aligns perfectly with the GPU’s memory controller, ensuring Coalesced Access.
